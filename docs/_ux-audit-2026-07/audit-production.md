## 2) ยืนยัน + ขยายผล Gate C5 (ROADMAP.md:52)

### 2.1 ✅ ยืนยัน: หน้าใบผลิตไม่โชว์ภาพลาย + ตารางไซส์

- **หลักฐาน server**: `production.getById` (src/server/routers/production.ts:97-122) select เฉพาะ order {orderNumber,title,deadline,priority,internalStatus,customer,items:{totalQuantity}} — **ไม่มี designs · ไม่มี variants/ไซส์ · ไม่มี prints (ตำแหน่ง/ขนาดลาย)**
- **หลักฐาน UI**: /production/[id]/page.tsx ทั้งไฟล์ (208 บรรทัด) ไม่มี `<img>`/design ใดๆ — มีแค่แถบบริบท (:128-157) + ขั้นตอน + เบิกเสื้อ + วัตถุดิบ
- **ความย้อนแย้ง**: ใบ job ticket ฉบับพิมพ์**มีครบแล้ว** — ภาพแบบอนุมัติ (print/job-ticket/[id]/page.tsx:163-185) + ตารางไซส์ต่อสินค้า (:227-243) + ตารางลายพิมพ์ต่อตำแหน่งพร้อมรูป (:246-289) → ข้อมูล+pattern query มีพร้อม แค่ยังไม่ถูกยกขึ้นหน้าจอ = ช่างวันนี้ต้องพึ่ง**กระดาษ**หรือกดปุ่ม "ใบสั่งงาน" เปิด print view (ไม่ mobile-friendly) หรือเข้า /orders/[id]
- **แก้ (surgical)**: เพิ่ม select ใน getById (designs approved take 1 {thumbnailUrl,fileUrl,versionNumber} + items→products→variants + items→prints) แล้วเพิ่มการ์ด "แบบ + ไซส์" บน /production/[id] เหนือรายการขั้นตอน — เลียนโครง job ticket เป๊ะ ไม่มีเงินในทุก field ที่เพิ่ม

### 2.2 ✅ ยืนยัน + เจอสาเหตุลึกกว่า: ปิดขั้น 5 แตะ

- **นับแตะจริง**จากหน้าใบผลิต: แตะแถวขั้นตอน (production-steps-list.tsx:122 — คลิกทั้งแถวเปิด dialog) → dialog StepUpdateDialog → แตะ dropdown สถานะ → เลือก "เสร็จแล้ว" → แตะบันทึก = **4-5 แตะ + รอ dialog โหลด** ต่อการปิด 1 ขั้น
- **สาเหตุลึก**: StepUpdateDialog (step-update-dialog.tsx:90-224) เป็นฟอร์มรวม 6 เรื่องในจอเดียว — สถานะ + ทำแล้ว/ทั้งหมด + ผู้รับผิดชอบ (หัวหน้า) + QC ผ่าน/ไม่ผ่าน + หมายเหตุ QC + หมายเหตุ → งานที่ช่างทำ 95% (แค่เปลี่ยนสถานะ/บอกจำนวน) จมอยู่ในฟอร์มของหัวหน้า
- **หมายเหตุสำคัญ**: บนบอร์ดเลน /production ปิดขั้นได้ **1 แตะอยู่แล้ว** (page.tsx:915-935 ปุ่มเริ่มทำ/เสร็จขั้นนี้) — C5 ข้อนี้เจ็บเฉพาะหน้าใบผลิต /production/[id] ซึ่งเป็นหน้าที่จอเช้า my-tasks ลิงก์เข้า (my-tasks/page.tsx:204) → ช่างที่เข้าจากจอเช้าเจอทาง 5 แตะเสมอ
- **แก้ (2 ทาง — เสนอ ก)**:
  - **ก. เพิ่มปุ่มเร็วบนแถว**: StepRow ใส่ปุ่ม "เริ่ม/เสร็จ" (h-11) แบบเดียวกับ LaneCardView + ถ้าขั้นนับจำนวน แตะ "เสร็จ" แล้วถาม 1 จอเล็ก (bottom sheet: จำนวนที่ทำ default = ที่เหลือ → ยืนยัน) = 1-2 แตะ · dialog เต็มยังอยู่หลังปุ่ม "เพิ่มเติม" สำหรับหัวหน้า (มอบงาน/QC/หมายเหตุ) — trade-off: แถวแน่นขึ้นเล็กน้อยบนจอแคบ
  - **ข. แยก dialog สองโหมด** (ช่างเห็นฟอร์มสั้น สถานะ+จำนวน · หัวหน้าเห็นเต็ม): แตะเท่าเดิม (ยังต้องเปิด dialog) แค่รกน้อยลง — ไม่ตอบโจทย์ "1-2 แตะ" จริง → ไม่แนะนำเป็นทางหลัก แต่ผสมกับ ก. ได้ (ฟอร์ม "เพิ่มเติม" ของช่าง = สั้น)
- **ของแถมที่ต้องเก็บด้วย**: dialog เดิมมี dropdown QC ผ่าน/ไม่ผ่าน (step-update-dialog.tsx:165-178) ซ้อนกับระบบใบนับ QC จริง (qc.create) — ช่างมีที่ให้ "ติ๊ก QC" สองที่ที่ความหมายไม่เท่ากัน → เสนอซ่อน QC ต่อขั้นจากฟอร์มช่าง เหลือไว้ในโหมดหัวหน้า (server updateStep รองรับอยู่แล้ว ไม่แตะ logic — production-step-plan.ts คุม QC_FAILED override ที่ supervise แล้ว)

## 3) จุดรก/จุดผิดที่เจอเพิ่ม (นอกเหนือ C5)

### 3.1 ปุ่มหลอกขัด B8 — print-runs + films โชว์ปุ่ม mutation ให้ทุก role

## 5) ข้อเสนอ "จอช่างที่ดี" ต่อจุดงาน + ทีวีคิวรวม

หลักคิด: **ไม่สร้างแอป/route กลุ่มใหม่ให้จอทัช** — จอทัช/มือถือช่างคือหน้าเดิมที่เติมข้อมูล ณ จุดงานให้ครบ (ช่าง login ปกติ สิทธิ์ PERM เดินตามคน · status ทุกปุ่มวิ่ง updateStep/isValidTransition server เดิม ไม่มีทางลัดใหม่) · มีของใหม่จริงแค่ **ทีวี read-only 1 หน้า**

### 5.1 จุดพิมพ์ (DTF printer) — เสริม /production/print-runs
- ทำ §2.3 (thumbnail + ลิงก์ไฟล์ + เวอร์ชัน ทุกแถวคิว/ทุกงานในรอบ)
- ปุ่มหลักใหญ่อยู่แล้ว (h-11) — เพิ่มเฉพาะ: แถวคิวโชว์ chip "รอฟิล์มเผื่อในคลัง n ชิ้น" ถ้าลายเดียวกันมีใน filmStock (กันพิมพ์ซ้ำ — ตอนนี้เตือนอยู่แค่ฟอร์มเปิดงาน) — *optional, ถามเบสก่อน (ต้อง match ลาย↔stock ซึ่งวันนี้ผูกด้วย order เท่านั้น)*
- ซ่อนปุ่ม mutation จาก role ที่ไม่มีสิทธิ์ (§3.1)

### 5.2 จุดรีด (heat press) — การ์ดรีด + stepper
- การ์ดเลน DTF ขั้น HEAT_PRESS: เพิ่ม **thumbnail ลาย + ตำแหน่ง/ขนาดพิมพ์** (kanban select เพิ่ม designs/prints — production.ts:194-243) — ช่างรีดเห็น "ลายอะไร รีดตรงไหน" โดยไม่เปิดอะไรเพิ่ม
- ปุ่ม **"+ บันทึกจำนวน"** ข้างปุ่มเสร็จ บนการ์ดที่ qtyTotal มีค่า → bottom sheet ตัวเลขใหญ่ + ปุ่ม +10/+50/ครบ (ยิง updateStep qtyDone เดิม) = อัปเดตบางส่วน 2 แตะ แทน 5 แตะ+เปลี่ยนหน้า (แก้ §3.4)
- ทางเลือกที่ตัดทิ้ง: หน้า /production/press แยก — ไม่คุ้ม (เลน DTF บนแท็บทำหน้าที่นี้แล้ว · เพิ่มหน้า = เพิ่มที่ให้หลง)

### 5.3 จุดแพ็ค — การ์ด PACK + ใบแพ็ค
- การ์ดเลน PACK + คอลัมน์กำลังแพ็ค: เพิ่มบรรทัดสรุปไซส์ (S×10 M×20 …) + ปุ่ม "ใบแพ็ค" → /print/packing-list/[id] (มีอยู่แล้ว ไม่มีเงิน)
- ธง blind ship คงเดิม (ดีแล้ว)

### 5.4 ทีวีคิวรวม (read-only) — ของใหม่ชิ้นเดียว
- **หน้าใหม่** `/(print)` style standalone: บอร์ดตัวหนังสือใหญ่ 4 คอลัมน์ **คิวพิมพ์ · คิวรีด (พร้อม/รอของ) · ตรวจ/แพ็ค · เลยกำหนด(แดง)** — เลขออเดอร์+ชื่องาน+จำนวน+กำหนดส่ง+ชื่อคนรับงาน · **ห้ามมี: เงินทุกชนิด, ชื่อลูกค้า (นโยบาย — จอเปิดโล่งในโรงงาน คนนอก/ช่างเห็นหมด → เสนอไม่ใส่, ถามเบส), ปุ่มใดๆ** · refresh ตัวเอง (polling 60s — ไม่ต้อง websocket, ทีม 5 คน) · dark bg ตัวอักษรใหญ่อ่านไกล 5 เมตร
- **endpoint ใหม่** `factoryBoard.queue` — select รายชื่อ field ตายตัว (ไม่ spread order) ให้ fail-closed เรื่องเงินตั้งแต่ shape · ข้อมูล derive จากของเดิม (getPrintQueue + kanban lanes + POST columns) ไม่มี logic ใหม่
- **auth 2 ทาง (ตัดสินยาก — เสนอ ข.)**:
  - **ก. บัญชี kiosk**: สร้าง user "จอโรงงาน" role PRODUCTION_STAFF, login ค้างบนทีวี, หน้าเป็น protectedProcedure ปกติ — *ดี: ไม่มีผิว auth ใหม่เลย · เสีย: Supabase session หมดอายุ/ทีวีดับแล้วต้องหยิบคีย์บอร์ดมา login ใหม่ + เป็นบัญชีผีใน users*
  - **ข. token จอ (แนะนำ)**: pattern เดียวกับ `/status`+`/job/<token>` ที่มีอยู่ (crypto.randomBytes + fail-closed + หมุนได้จาก Settings) — URL `/tv/<token>` เปิดครั้งเดียวอยู่ยาว ทีวีรีบูตก็รอด — *ดี: เหมาะ kiosk จริง, repo มี pattern พร้อม · เสีย: เพิ่มผิว public 1 จุด (แต่ endpoint ไม่มีเงิน/ไม่มี mutation โดย shape จึงเสี่ยงต่ำ + rate-limit firewall ครอบ path ได้แบบ B15)*
