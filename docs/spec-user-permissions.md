## 1) สิ่งที่มีอยู่แล้ว (ไม่ต้องสร้างใหม่)

- **หน้าจัดการพนักงานมีแล้ว ใช้ได้วันนี้**: `Settings → ผู้ใช้` (`/settings/users`) —
  เพิ่มพนักงาน (ชื่อ/อีเมล/ตำแหน่ง/รหัสผ่านชั่วคราว) · เปลี่ยนตำแหน่ง · เปิด-ปิดบัญชี
  (ปิดแล้วโดน ban ฝั่ง Supabase ด้วย) · รีเซ็ตรหัสผ่าน — ทั้งหมดจำกัด OWNER + ลง audit log
  - เบื้องหลังใช้ Supabase admin API (`user.create` ใน `src/server/routers/user.ts:52-121`
    สร้าง auth user + แถวในตาราง users ใน transaction พร้อม rollback)
- **ตำแหน่ง (role) 6 ตัว**: OWNER · MANAGER · ACCOUNTANT · SALES · PRODUCTION_STAFF · DESIGNER
- **ด่านสิทธิ์ปัจจุบัน**: ผูก role ตายตัวในโค้ด รวม **183 จุด** — server 111 (requireRole) +
  client 60 (เมนู/ปุ่ม/คอลัมน์เงิน) + เทียบ role ตรง 12 + ชุดสถานะต่อ role 3 ชุด
  (เพิ่งผ่าน security review ⑦ + moneygate 29 เช็คกับ DB จริง)

**ช่องว่างเดียวที่เหลือ** = ปรับสิทธิ์ "รายคน" ไม่ได้ — ทุกอย่างผูกตำแหน่ง อยากให้ช่างคนหนึ่ง
เห็นราคาขายแต่อีกคนไม่เห็น ต้องแก้โค้ด

## 2) รายการสิทธิ์ของระบบ (catalog ร่าง — 20 ตัว ครอบทั้ง 183 จุด)

กลุ่มเงิน/เอกสาร (โซนอ่อนไหว — เฟสแรก):
1. `see_order_money` เห็นเงินฝั่งขาย (ราคา/ยอดออเดอร์/บิล/ยอดลูกค้า) — ~30+ จุด
2. `see_finance` เห็นทุน/กำไร/รายงานการเงิน — ~9 จุด
3. `record_payments` บันทึกรับเงิน/คืนเงิน/ยกเลิกบิล — ~12 จุด
4. `manage_billing_documents` ออกใบแจ้งหนี้/ใบวางบิล/ใบลดหนี้ — ~19 จุด
5. `create_sales_docs` เปิด/แก้ออเดอร์+ใบเสนอ — ~27 จุด
6. `manage_customer_data` แก้ข้อมูลลูกค้า/วงเงินเครดิต — ~10 จุด
7. `view_analytics` ดู dashboard สถิติ — ~5 จุด

กลุ่มปฏิบัติการ (เฟสสอง — ความเสี่ยงต่ำกว่า):

ใบเสนอ (ยืนยัน/แปลง) · outsource · ต้นทุนวัสดุ · แก้ออเดอร์หลังอนุมัติ (change order)

## 3) ทางเลือก (เทียบตรงๆ)

### A. ใช้ระบบ role เดิมต่อ (ไม่สร้างอะไรใหม่) — 0 วัน
- ทำได้ทันที: เพิ่มพนักงาน+เลือก 1 ใน 6 ตำแหน่งที่ /settings/users
- ✅ ฟรี ไม่มีความเสี่ยง · ❌ ปรับรายคนไม่ได้ — เจอเคสแบบ "ทุนสินค้าบนแคตตาล็อก" ต้องแก้โค้ดทุกครั้ง

### B. role เดิมเป็นค่าเริ่มต้น + ติ๊กปรับสิทธิ์รายคนได้ (แนะนำ) — เฟสแรก ~3-4 วันทำงาน
- **หลักคิด**: ตำแหน่ง = ชุดสิทธิ์เริ่มต้น (ตรงพฤติกรรมระบบวันนี้เป๊ะ) → หน้าผู้ใช้เพิ่มปุ่ม
  "สิทธิ์" ต่อคน → checklist ติ๊กเพิ่ม/ตัดสิทธิ์เฉพาะคนนั้น (ตรงกับที่เบสอธิบายมากที่สุด
  โดยไม่ต้องรื้อ concept ตำแหน่งทิ้ง — ทีม 5 คนยังคุยกันด้วยคำว่า "ช่าง/บัญชี/ขาย" ได้)
- **เทคนิค**: เพิ่มช่อง `permissionOverrides` (JSON) บนตาราง users + helper กลาง
  `hasPermission(user, "...")` — จุดเช็คทั้งระบบค่อยๆ เปลี่ยนมาเรียกตัวนี้ (default จาก role
  เดิม = วันแรก **พฤติกรรมไม่เปลี่ยนเลย** จนกว่าจะมีคนไปติ๊ก override)
- **เฟส 1** (~3-4 วัน): โครง schema+helper+UI ติ๊กสิทธิ์ + migrate กลุ่มเงิน/เอกสาร (ข้อ 1-8)
  ~70 จุดสำคัญ + เขียน moneygate เพิ่ม (ทดสอบ override จริงกับ DB) + รัน verify ครบชุด
- **เฟส 2** (~2-3 วัน · แยกก้อน ทำทีหลังได้): กลุ่มปฏิบัติการ (ข้อ 10-20) + เก็บกวาด
  inline check 26 จุดฝั่งจอให้วิ่งผ่าน helper เดียวกัน
- ⚠️ ความเสี่ยง: แตะด่านเงินที่เพิ่งปิดแน่น — ต้อง re-verify moneygate + ชุด DB จริงทั้งหมด
  ทุกเฟส (แผน verify อยู่ข้อ 5) · MCP/agent key ใช้ role เดิม ต้องให้วิ่งผ่าน helper ด้วย

### C. ระบบสิทธิ์เต็มรูป (สร้าง role เองได้ + matrix อิสระ) — ~2 สัปดาห์+
- สร้างตำแหน่งใหม่เองได้ กำหนดสิทธิ์ทุกตำแหน่งอิสระ ตาราง role-permission ใน DB
- ❌ สำหรับทีม 5 คน = เกินความจำเป็น (over-engineer ตามกติกา build ข้อ 3) · แรง+เสี่ยงสูงสุด ·
  ทุกอย่างที่ C ทำได้ B ทำได้ที่ระดับรายคนอยู่แล้ว — เขียนไว้ให้เห็นภาพ ไม่แนะนำ

## 4) สิ่งที่ตั้งใจ "ไม่ทำ" (ทุกทางเลือก)
- สิทธิ์ละเอียดกว่า catalog (รายปุ่ม/รายฟิลด์) — จอจะกลายเป็น switchboard · catalog 20 ตัวพอ
- custom role builder (ดูข้อ C) · สิทธิ์ตามช่วงเวลา/ตามลูกค้า — ยังไม่มีเคสจริง
- แตะ `isValidTransition`/state machine — สิทธิ์คุมว่า "ใครกด" ไม่ใช่ "เส้นทางไหนถูก"
